\section{Analysis and description of collaborative local-first applications}

% Describe the problem and analyze its context here.
Collaborative applications have not always been so popular as they are these days. You might not 
even know that some of applications that we use every day are collaborative. In everyday life of a 
software developer almost all of modern IDEs have collaborative work options, almost all project management 
software are also collaborative. These are examples close to software engineers, but these applications do 
not limit themselves to software development. Google Docs, Microsoft 365, Figma, Miro and many more are examples of
collaborative applications that are used by millions of people every day. These applications are not only used for work, 
but also for education and entertainment. 

It hasn't always been like this. The steep rise of collaborative applications began as the remote work became 
more popular but also it owes its rise of popularity to universality of fast global network and increase of computing power
of everyday devices. 

Current collaborative apps couldn't exist 20 years ago. They haven't been simply invented yet. Typical 
application of early days of online applications had a simple architecture. It was a client-server application. 
The server was responsible for storing data and processing it, while the client was responsible for presenting the 
data to the user and sending user input to the server. 
This architecture had its advantages and disadvantages. It was simple and easy to implement, but it had a single point of
failure.
If the server went down, the entire application went down. It also had scalability issues, as the server had to handle all the requests from the clients.

As we can see the level of complication if very high. Simple user intercation with a system 
triggers multiple functions from multiple modules from multiulpe services. This level of 
sophistication simply cannot be used for collaborative applications, because orchestration 
of synchoronization of state shown in simultaneus users would be extremely difficult. 

For one, in would require to many asynchronous time-consuming remote calls to different services. 
Total synchronization nightmare. For some time tough there has been efforts to implement collaboration 
this way. This approach was known as OT - operational transformation (still used in Google Docs). Although Google Docs is a 
good example of usage of cloud-first approach, in this thesis I decided to focus on local-first architecutres
as they have proved to be more broadly used in the industry and it offers way more possibilities of different

In simplest words the definition of cloud-approach is "If the interaction of the user with the system haven't been

\section{OT vs CRDTs}

One very important topic in the matter of conflict resolution and collaboration is data structures algorightms 
that can be used. Local / remote inserts into data structures need to be \textbf{commutative}, \textbf{associative} and \textbf{idempotent}.
These three properties are required to ensure that all replicas of data will be eventually consistent - assuming that all 
operations (inserts/deletes) propagate properly. There are two main
approaches to achieve this - OT (operational transformation) and CRDTs (conflict-free replicated data types).

Historically well known solution is OT - operational transformations. The basic idea
behind this approach is to transform operations in such a way that they can be applied in any order and still produce the same result.
This approach was used in Google Docs and it is still used in some applications. However, it has some drawbacks. It is complex to implement and
 it can lead to conflicts that are difficult to resolve.

CRDTs - conflict-free replicated data types - are a more recent approach to achieve eventual consistency in distributed systems.
The basic idea behind CRDTs is to design data structures in such a way that they can be updated independently and still converge
 to the same state. The algorithms are implemented directly into data structures. This approach is simpler to implement and it can
  handle conflicts more gracefully. It is used in many modern applications, including Figma, Miro, Notion, Obsidian, Roam Research, 
  Coda and many more.

\section{Vision of the System}

The envisioned system is a desktop application designed for real-time collaborative drawing on a whiteboard.
 It leverages the Rust programming language for performance and reliability, and utilizes the egui library 
 to provide a modern, responsive user interface. Communication between users is achieved through WebRTC,
  with the Livekit open-source server facilitating room-based connections. Each room acts as a channel 
  or document, allowing users to join by simply entering a room code or name, without the need for user 
  accounts, authentication, or authorization.

The application employs CRDTs (Conflict-Free Replicated Data Types) from the Automerge library to ensure 
seamless synchronization of whiteboard state across all participants. This approach enables users to draw
 and erase simultaneously, with changes instantly reflected for everyone in the room. The cursors of other
   users are visible, enhancing the collaborative experience. The whiteboard will be represented as a canvas 
   with raster graphics of strokes from users as vector of points with certain width and color.

For real-time, multithreaded communication, the system uses the Tokio library, ensuring efficient handling
 of concurrent operations. Notably, the application does not persist data by default; once the app is 
 closed, all data is lost unless explicitly exported to a file by the user. The focus is on desktop
  environments, with no mobile version planned.

\section{System Architecture}

As mentioned in the introduction, the system is designed with a star topology architecture, where a central 
server (open source LiveKit SFU) facilitates communication between clients. Each client maintains a local
 CRDT document representing the whiteboard state, and synchronization occurs through WebRTC data channels.
  The architecture is modular, with clear separation of concerns between the user interface (egui), the 
  CRDT data model (Automerge), and the networking layer (LiveKit/Tokio).

The program is devided mostly into three main modules: \texttt{ui.rs} which holds most of the application interface
state and UX interaction methods, \texttt{backend\_api.rs} -- a module of abstraction which defines connection points 
of canvas/CRDT and networking, because what user sees on the screen is not just a matrix of pixels from 
persistent storage, but a result of rendering of the CRDT document.

\texttt{automerge\_backend.rs} -- a module which defines the CRDT document and methods of interaction with it. It is 
an implementation of \texttt{backend\_api.rs} with a little bit of extra methods.

\subsection{State-based vs Operation-based synchronization}
%  brief theoretical background, position Automerge as as state-based CRDT 
%  with operation-log hisotry

Automerge library is positioned as a state-based CRDT because it uses a specialised 
synchronization protocol to find differences between peers.

\begin{itemize}
	\item \textbf{Sync States}: each peer maintains a \texttt{sync::State} for every other connected peer.
	This acts as a tracker for what the remote peer already knows about the document.
	\item \textbf{Delta updates}: instead of sending the entire document, the backend uses 
	\texttt{generate\_sync\_message()} to produce a compact binary blob containing only the changes 
	(deltas) the specific remote peer has not seen yet.
	\item \textbf{Convergence through merging}: when a peer receives this message, the \texttt{receive\_sync\_message}
	function decodes the data and automatically merges the remote changes into a local 
	document.
\end{itemize}

Figure below illustrates the synchronization sequance between two peers - A and B.
 When A makes a local change, it generates a sync message containing only the new
 operations that B has not seen yet. This message is sent to B, which applies it 
 to its local document. If B has made concurrent changes, the merge process ensures 
 that both A and B converge to the same final state without conflicts.

\subsection{Automerge sync protocol}

The synchronization process in Automerge is a state-based handshake designed to bring two peers
into the same identical state by exchanging only the necessary data. This is managed 
through a pair of functions: \texttt{generate\_sync\_message} and 
\texttt{receive\_sync\_message}. 

% https://arxiv.org/abs/2012.00472
\begin{enumerate}
	\item \textbf{The Handshake: \texttt{generate\_sync\_message} and \texttt{receive\_sync\_message} }
		\\ The handshake functions as a continuous conversation between peers until they are fully 
		synchronized.
		\begin{itemize}
			\item \verb|generate_sync_message(peer_id)|: This function asks Automerge 
			to identify what has changed since the last time this specific peer was updated. 
			It uses a peer-peer \verb|sync::State| to track progress. If there are new 
			changes (deltas), they are packed into a binary blob; if the peer is already 
			up to date, it returns \verb|None|.
			\item \verb|receive_sync_message(peer_id, message)|: When a peer receives this 
			binary blob, it "unpacks" the changes and merges them into its own document.
			This function also updates the local \verb|sync::State| for that sender, marking 
			those specific changes as "received" so they aren't requested again. 
		\end{itemize}
	\item \textbf{The role of bloom filters} \\
		Automerge uses Bloom filters to efficiently track "what the peer already knows"
		without having to send a massive list of every operation ever performed. 
		\begin{itemize}
			\item Efficient summarization: Instead of sending full history, a peer sends
			a compact Bloom filter---a probabilistic data structure---that represents the set 
			of operations (changes) it currently holds in its document. 
			\item Identify gaps: when receiving, the peer compares its own document against 
			this bloom filter, it can quickly identify which operations it has that are not 
			represented in the filter. These missing pieces are then bundled into the next 
			sync message.
		\end{itemize}
	\item \textbf{Why multiple rounds may be needed} \\
		While a single change might sync in one go, a full synchronization often requires 
		two to three rounds to reach convergence.
		\begin{itemize}
			\item \textbf{Round 1 (Discovery):} Peer A sends a sync message containing its current "heads" (hashes of recent changes) and a Bloom filter summarizing all the changes it has.
			\item \textbf{Round 2 (Response):} Peer B receives this, uses the Bloom filter to identify which of its own changes Peer A is missing, and sends those changes back. Peer B also sends its own Bloom filter.
			\item \textbf{Round 3 (Convergence):} Peer A applies the changes from Peer B. It now knows if it is still missing anything based on Peer B's Bloom filter. It sends any remaining changes Peer B might need.
			\item \textbf{Round 4 (Finalization):} Peer B applies those changes. At this point, Peer B generates a final "empty" sync message to confirm to Peer A that they are now in identical states.
		\end{itemize}
\end{enumerate}

Synchronization often requires multiple round-trips to reach convergence, especially if the 
divergence between peers is significant. The process mimics a conversation where peers exchange 
what they know until no new information remains. The following Rust code snippet illustrates this 
iterative negotiation pattern used during a sync session:

\begin{verbatim}
let mut peer1_state = sync::State::new();
let mut peer2_state = sync::State::new();

loop {
    // 1. Peer 1 generates a message for Peer 2
    let msg1 = doc1.sync().generate_sync_message(&mut peer1_state);
    
    if let Some(m) = msg1 {
        // Peer 2 applies changes from Peer 1
        doc2.sync().receive_sync_message(&mut peer2_state, m)?;
    }

    // 2. Peer 2 generates a message for Peer 1
    let msg2 = doc2.sync().generate_sync_message(&mut peer2_state);
    
    if let Some(m) = msg2 {
        // Peer 1 applies changes from Peer 2
        doc1.sync().receive_sync_message(&mut peer1_state, m)?;
    }

    // Stop when both sides have nothing left to send (convergence reached)
    if msg1.is_none() && msg2.is_none() {
        break;
    }
}
\end{verbatim}

This loop ensures that even if one message doesn't carry all necessary changes (due to size limits 
or dependency resolution), subsequent messages will complete the synchronization.

\subsection{Conflict resolution and guarantees}

In Automerge (and most CRDTs), conflict resolution is governed by the principle of 
\textbf{strong eventual consistency}. This means that if two peers perform different actions 
on the same piece of data at the same time, they must both arrive at the exact same result 
without a central server deciding the outcome. Automerge achieves this through deterministic 
rules embedded directly into the data structure operations.

\subsubsection*{The ``Clear'' operation (Delete vs.\ Edit)}

In Automerge, a delete operation (such as clearing a list or removing an element) specifically 
targets the set of \emph{visible Change IDs} that the peer has observed at the moment the 
operation is issued.

\begin{itemize}
    \item \textbf{Scenario}: Peer~A clears the canvas (deletes all strokes from the list). 
    Simultaneously, Peer~B draws a new stroke.
    \item \textbf{Logic}: Peer~A's ``Clear'' command only knows about the strokes that existed 
    at the moment Peer~A clicked clear. It has no knowledge of the new stroke Peer~B is 
    currently creating, because that stroke has not yet been synchronized.
    \item \textbf{Result}: The new stroke from Peer~B \emph{survives}. The old strokes are 
    deleted (since both A and B agree they existed), but the new stroke remains because 
    Peer~A's delete operation did not ``reach'' it. This is the \textbf{Add wins} semantics 
    --- insertions that a peer has not observed cannot be removed by that peer's concurrent 
    delete.
\end{itemize}

\subsubsection*{The ``Draw'' operation (Concurrent edits to the same property)}

If both peers modify the same property concurrently (for example, changing the color of the 
same stroke at the same time - hypothetically, because 1.0.0 version of application does not allow 
such operations), Automerge must deterministically choose which value to display.

\begin{itemize}
    \item \textbf{Scenario}: Peer~A sets a stroke color to Red. Peer~B sets the same stroke 
    color to Blue.
    \item \textbf{Logic}: Automerge performs a deterministic \textbf{Last Write Wins (LWW)} 
    tie-break based on a lexicographical comparison of the internal Change IDs (hashes).
    \item \textbf{Result}: If Peer~A's change hash is \texttt{0xabc...} and Peer~B's is 
    \texttt{0xdef...}, Peer~B wins because its hash is lexicographically higher. Crucially, 
    both peers will see the same winner. No data is actually lost in the operation history 
    --- Automerge preserves the full history --- but only one value is displayed as the 
    ``current'' state.
\end{itemize}

The following table summarizes the conflict resolution strategies:

\begin{table}[H]
    \centering
    \caption{Conflict resolution strategies in Automerge}
    \begin{tabular}{|p{0.12\textwidth}|p{0.25\textwidth}|p{0.50\textwidth}|}
        \hline
        \textbf{Action} & \textbf{Conflict Type} & \textbf{Resolution} \\
        \hline
        Clear & Add vs.\ Remove & Add wins. Deletes only affect elements the peer 
        observed at the time of deletion. \\
        \hline
        Draw & Value vs.\ Value & Deterministic tie-break. The change with the higher 
        lexicographical hash becomes the visible value. \\
        \hline

\subsubsection*{Implications for the whiteboard application}

In the implemented Rust application, strokes are stored in an Automerge \texttt{List}. 
Calling \texttt{splice} to clear the list or \texttt{insert} to add a stroke is inherently 
safe under concurrent usage. If one user clears the whiteboard while another draws a new 
stroke, that new stroke will appear on both screens after the sync loop finishes, while the 
old ``cleared'' strokes will vanish. This prevents the ``lost update'' problem where one 
user's work is accidentally wiped out by another's cleanup action.

If the application-level intent is that ``Clear'' should mean ``wipe everything, including 
things I have not yet seen,'' this must be handled at the application layer --- for example, 
by storing a \texttt{last\_cleared\_timestamp} and filtering out any strokes created before 
that time. In the current implementation, the simpler CRDT-native semantics (add wins) are 
used, which is the safer and more predictable default for collaborative drawing.


\subsection{Initial synchronization of a new peer}

A critical aspect of the collaborative experience is the ability for a new user to join an 
existing session and seamlessly acquire the current state of the whiteboard. 
When a new peer joins a room (mid-session), they start with an empty local document (or a 
stale version if they were previously connected). The synchronization protocol handles this 
case naturally without special "initialization" logic.

Upon connection, the existing peers in the room detect the new participant (via LiveKit events) 
and initiate the standard sync handshake described in Section~2.4.2. Because the new peer's 
knowledge state is zero (empty Bloom filter), the first sync message generated by an existing 
peer will contain the entire history of operations needed to reconstruct the current document state. 
The new peer receives this "full" update, applies it, and instantly converges to the same state 
as the rest of the group. This process ensures that late-joining users are never out of sync 
and have access to the full drawing history, allowing them to undo or modify strokes made before 
they arrived.

\section{CRDT Data Model -- What is being synchronized?}

Having described \emph{how} the synchronization protocol achieves consistency, this section 
focuses on \emph{what} is being synchronized --- the concrete data structures stored inside 
the Automerge document. The data model defines how user-visible whiteboard elements (strokes, 
background) are represented as CRDT objects, how user actions map to CRDT operations, and how 
the entire document can be persisted to and restored from disk.

\subsection{Document schema}

The Automerge document follows a simple, flat schema with two top-level keys stored under the 
document root (\texttt{ROOT}):

\begin{verbatim}
ROOT
 +-- "strokes"    : List<String>       // JSON-serialized strokes
 +-- "background" : Bytes              // optional background image
\end{verbatim}

\begin{itemize}
    \item \textbf{\texttt{strokes}} --- an Automerge \texttt{List} where each element is a JSON 
    string representing a single \texttt{Stroke} object. The list is created lazily: if it does 
    not exist when a stroke is drawn, the backend creates it with 
    \texttt{put\_object(ROOT, "strokes", ObjType::List)}.
    
    \item \textbf{\texttt{background}} --- an optional \texttt{Bytes} scalar stored directly at 
    the root. It holds the raw binary data (e.g., PNG) of a background image, if one has been 
    loaded by the user.
\end{itemize}

\paragraph{Why JSON strings inside a List?}
An alternative design would be to use native Automerge maps and nested lists to represent each 
stroke's points, color, and width. However, storing strokes as opaque JSON strings offers 
several practical advantages:

\begin{enumerate}
    \item \textbf{Simplicity}: Serialization and deserialization are handled by \texttt{serde\_json}, 
    avoiding the complexity of managing deeply nested Automerge objects with multiple object IDs.
    \item \textbf{Atomicity}: Each stroke is inserted or removed as a single list element. There is 
    no risk of partially synchronized strokes (e.g., points arriving without their color).
    \item \textbf{Performance}: Fewer CRDT metadata objects are created, reducing memory overhead 
    and sync message size.
    \item \textbf{Trade-off}: The downside is that individual stroke properties (e.g., color) cannot 
    be edited in-place without replacing the entire JSON string. For the whiteboard use case, where 
    strokes are immutable once drawn, this trade-off is acceptable. A notable consequence of this 
    design is the eraser implementation: rather than removing or modifying existing strokes, the 
    eraser tool works by drawing new strokes with a white color (\texttt{[255, 255, 255, 255]}), 
    effectively painting over previous content. This approach is discussed further in the 
    ``Eraser implementation'' paragraph below.
\end{enumerate}

\subsection{Stroke representation}

The data types that model a stroke are defined in the \texttt{backend\_api} module. 
A \texttt{Stroke} consists of three fields:

\begin{verbatim}
pub struct Point {
    pub x: i32,
    pub y: i32,
}

pub struct Stroke {
    pub points: Vec<Point>,    // ordered sequence of coordinates
    pub color: [u8; 4],        // RGBA color
    pub width: f32,            // line thickness in pixels
}
\end{verbatim}

\paragraph{Why a vector of points rather than pixels?}
Two approaches exist for representing drawn content: storing raw pixel data (a bitmap) or 
storing geometric primitives (vectors of points). The vector approach was chosen for several 
reasons:

\begin{itemize}
    \item \textbf{Compactness}: A stroke of 200 sampled points requires approximately 
    $200 \times 8 = 1600$~bytes plus color and width metadata. A rasterized equivalent on a 
    $1920 \times 1080$ canvas would require megabytes.
    \item \textbf{Resolution independence}: Strokes can be re-rendered at any zoom level or 
    canvas size without loss of quality.
    \item \textbf{CRDT friendliness}: Each stroke is a self-contained, serializable unit that 
    can be inserted into or removed from the Automerge list atomically. Bitmap-based approaches 
    would require pixel-level conflict resolution, which is far more complex.
    \item \textbf{Undo granularity}: Removing a stroke means deleting one list element rather 
    than restoring thousands of pixels.
\end{itemize}

\subsection{Intent abstraction}

User actions are abstracted through the \texttt{Intent} enum, which decouples the UI from 
the CRDT operations:

\begin{verbatim}
pub enum Intent {
    Draw(Stroke),   // add a new stroke to the document
    Clear,          // remove all strokes from the document
}
\end{verbatim}

Each intent maps directly to specific Automerge CRDT operations:

string at the end, and finally reads back all strokes to produce a \texttt{FrontendUpdate} for 
the UI.

For \texttt{Intent::Clear}, the backend retrieves the list and calls \texttt{splice} with a 
delete count equal to the list length and an empty iterator for replacements. This removes all 
elements in a single CRDT operation. As discussed in Section~2.4.3, this delete only affects 
strokes that the local peer has observed --- any concurrently inserted strokes from remote 
peers will survive the clear.

\paragraph{Eraser implementation}
Although the user interface presents the eraser as a distinct tool alongside the pen, the 
eraser does not correspond to a separate \texttt{Intent} variant. Instead, the eraser is 
implemented entirely at the UI layer by substituting the user-selected color with opaque white 
(\texttt{Color32::WHITE}, i.e.\ RGBA \texttt{[255,\,255,\,255,\,255]}). When the user 
releases the eraser stroke, the UI commits it through the same \texttt{Intent::Draw} path as 
a regular pen stroke. From the CRDT's perspective, an eraser action is indistinguishable from 
drawing a white line --- it is simply another entry appended to the \texttt{"strokes"} list.

This design has several implications:

\begin{itemize}
    \item \textbf{Simplicity}: No additional CRDT operation or intent type is needed. The 
    backend remains unaware of the eraser concept entirely, keeping the synchronization logic 
    minimal.
    \item \textbf{Additive nature}: Eraser strokes are additive --- they increase the size of 
    the stroke list rather than removing elements. Over extended drawing sessions, this can 
    lead to a growing document even when the user is ``erasing'' content.
    \item \textbf{Background limitation}: Because the eraser paints white pixels, it only 
    produces the expected visual effect on a white canvas. If a background image has been 
    loaded, the eraser will paint opaque white over the background rather than restoring the 
    underlying image.
    \item \textbf{No structural removal}: Individual strokes beneath the eraser path are not 
    removed from the CRDT document. They remain in the operation history and continue to be 
    synchronized with peers. The ``erasure'' is purely visual --- the white stroke is rendered 
    on top during the rasterization pass, hiding the strokes underneath.
\end{itemize}

An alternative approach would be to introduce an \texttt{Intent::Erase} variant that performs 
hit-testing against existing strokes and removes intersecting entries from the Automerge list 
using \texttt{splice}. This would produce true structural deletion and reduce document growth. 
However, it would require geometric intersection logic (testing whether the eraser brush 
overlaps each stroke's point sequence) and introduce more complex CRDT operations. For the 
current version of the application, the simpler white-stroke approach was chosen as a 
conscious trade-off favoring implementation simplicity over document efficiency.

\subsection{File persistence}

The application does not persist data by default. However, the user can explicitly save and 
load documents through the \texttt{save} and \texttt{load} methods of the \texttt{DocBackend} 
trait. Two export formats are supported:

\begin{itemize}
    \item \textbf{\texttt{.crdt} (full state)}: The \texttt{save()} method calls 
    \texttt{self.doc.save()}, which serializes the entire Automerge document --- including all 
    operation history, actor IDs, and the current state --- into a compact binary format. 
    Loading this file with \texttt{AutoCommit::load(\&data)} fully reconstructs the document, 
    preserving the ability to merge with other peers' documents. After loading, all sync states 
    are cleared because the relationship with previously connected peers is no longer valid.
    
    \item \textbf{\texttt{.png} (raster snapshot)}: The UI layer can also export the current 
    canvas as a PNG image. This is a one-way export --- the raster image cannot be loaded back 
    into the CRDT --- but it is useful for sharing the whiteboard content outside the 
    application.
\end{itemize}


\begin{verbatim}
// Saving: serialize the full CRDT document to bytes
fn save(&mut self) -> Vec<u8> {
    self.doc.save()
}

// Loading: reconstruct document from bytes, reset sync states
fn load(&mut self, data: Vec<u8>) {
    if let Ok(doc) = AutoCommit::load(&data) {
        self.doc = doc;
        self.sync_states.clear();
    }
}
\end{verbatim}

The \texttt{.crdt} format preserves the full operation history, which means that a loaded 
document can be taken into a new collaborative session and merged with other peers' changes 
seamlessly. This is a direct benefit of the CRDT architecture --- the document carries its 
own synchronization context.


\section{Communication Layer - Where the messages travel ?}

In previous sections, we have established \emph{how} the synchronization protocol achieves consistency and \emph{what} 
data structures are being synchronized. This section focuses on the \emph{communication layer} --- the transport
 infrastructure that carries sync messages and other real-time updates between peers in a collaborative session.
% - livekit room connection, data channel publishing/subscribing, broadcasting of messages/cursor positions,
% - usage of Tokio.
\subsection{LiveKit and WebRTC data channels}
% star topology via livekit SFU, room-basen sessions, why webRTC data channels
The application uses LiveKit as the underlying real-time communication platform. 
LiveKit provides a scalable SFU (Selective Forwarding Unit) architecture that 
allows multiple peers to connect to a shared room and exchange media and 
data streams efficiently. It is an open-source alternative to services like 
Twilio or Agora, offering low-latency connectivity and built-in support for 
WebRTC data channels, which are ideal for real-time synchronization of CRDT 
messages and cursor positions in a collaborative whiteboard application.
This solution uses a start topology where each peer connects to the LiveKit server,
 which forwards messages to all other participants in the room. The pros and 
 cons of this approach were discussed in the introduction chapter.


WebRTC data channels provide a low-latency, peer-to-peer communication
 mechanism that is ideal for transmitting arbitrary binary data between 
 clients in real time. Unlike traditional client-server HTTP APIs, data 
 channels are bidirectional and support both reliable and unreliable delivery 
 modes, making them suitable for collaborative applications where timely 
 updates are critical.

In this system, each client establishes a WebRTC data channel through the LiveKit 
server upon joining a room. All CRDT synchronization messages, cursor updates, and 
chat messages are serialized into binary form and sent over this channel. 

LiveKit abstracts the complexity of WebRTC signaling and connection management, 
so the application code can focus on encoding and decoding messages for 
synchronization. This approach provides a robust foundation for building scalable,
 responsive collaborative tools.


\subsection{Thread modeling}
% tokio runtime on a background thread, MPSC channels between 
% ui thread and networkj thread, 
Application in its sourcecode uses Tokio runtime library which enables 
convinient usage of threads in application. The UI runs on the main thread, 
while all network communication and CRDT synchronization logic is executed 
on a separate background thread managed by Tokio. This separation ensures that the user interface remains responsive and
 does not block while waiting for network messages or performing 
 potentially expensive CRDT operations.

 In rust communication between threads is handled using MPSC 
 (multi-producer, single-consumer) one-way channels. In this application, 
 the UI thread sends user intents (e.g., drawing a stroke) to the 
 backend thread via an MPSC channel. The backend processes these intents, 
 updates the CRDT document, and generates frontend updates 
 (e.g., the new list of strokes) that are sent back to the UI 
 thread through another MPSC channel. This asynchronous message-passing 
 architecture allows for smooth interaction without freezing the UI, 
 even during intensive synchronization tasks. But before this happens, 
 client needs to connect to the LiveKit room, which is also done on the 
 background thread to avoid blocking the UI during connection setup.

 to LiveKit room.

The flow traverses six participants that correspond to distinct runtime components: the remote peer, the LiveKit room (server-side), the background Tokio thread, the two ends of the MPSC channel (\texttt{UnboundedSender<AppMsg>} and \texttt{UnboundedReceiver<AppMsg>}), and the UI thread.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.99\textwidth]{./images/data-receiving-seq.png}
	\caption{Sequence diagram: Receiving and reassembling chunked data messages}
	\label{fig:data-receiving-seq}
\end{figure}

Knowing the data types we can now describe the message flow in detail. The process begins when a remote peer calls \texttt{publish\_data(payload)} on its local LiveKit participant. The LiveKit SFU forwards the payload to all other participants in the room, which triggers a \texttt{RoomEvent::DataReceived\{payload, participant\}} event on the background thread's event loop. The background thread immediately attempts to deserialize the raw bytes into a \texttt{TransportPacket} --- the wire-level envelope described earlier in this section.

At this point, the flow branches depending on the packet variant. If the packet is a \texttt{TransportPacket::Message}, the payload is small enough to have been sent in a single frame. The background thread deserializes the inner bytes directly into a \texttt{NetworkMessage} and forwards it to the UI thread by sending an \texttt{AppMsg::NetworkMessage\{sender, message\}} through the MPSC channel. If, however, the packet is a \texttt{TransportPacket::Chunk}, the message was too large for a single LiveKit frame and was split by the sender into multiple chunks. Each chunk carries a unique transfer identifier, its index within the sequence, and the total number of expected chunks. The background thread stores the incoming chunk in a per-peer \texttt{incomplete\_transfers} buffer --- a hash map keyed by the transfer identifier. After each chunk is stored, the thread checks whether all chunks for that transfer have arrived. If chunks are still missing, the thread simply returns to the event loop and waits for the next packet. Once the final chunk arrives and the buffer is complete, the thread concatenates all chunks in index order, reconstructs the full byte sequence, deserializes it into a \texttt{NetworkMessage}, and sends the result to the UI thread through the same MPSC channel path.

LiveKit imposes a maximum message size limit of approximately 14\,KB for data channel packets. To accommodate this constraint, the sending side implements a symmetric chunking mechanism: before publishing, it checks the serialized size of the \texttt{NetworkMessage}. If the payload fits within the 14\,KB threshold, it is wrapped in a \texttt{TransportPacket::Message} and sent as a single packet. Otherwise, the payload is divided into 14\,KB segments, each wrapped in a \texttt{TransportPacket::Chunk} with the appropriate metadata, and sent sequentially. A randomly generated 64-bit identifier ties all chunks of the same logical message together, ensuring that concurrent transfers from different peers or even from the same peer do not interfere with each other.

After the \texttt{AppMsg} is placed on the channel, the background thread calls \texttt{ctx.request\_repaint()} to wake up the egui rendering loop. On the next frame, the \texttt{update()} method of \texttt{AppView} polls the receiver with \texttt{try\_recv()} and retrieves the pending \texttt{AppMsg::NetworkMessage}. The UI then inspects the inner \texttt{NetworkMessage} variant and dispatches accordingly: a \texttt{Sync} message is forwarded to the CRDT backend via \texttt{receive\_sync\_message()}, which merges the remote changes and returns a \texttt{FrontendUpdate} that triggers a canvas redraw followed by a \texttt{sync\_with\_all()} call to propagate any resulting state to other peers; a \texttt{Chat} message is appended to the event log displayed in the LiveKit panel; and a \texttt{Cursor} message updates the sender's entry in the \texttt{remote\_cursors} map so that the next rendering pass draws the remote pointer at the correct position.

The entire message flow between the UI and the network layer is handled by four custom enum types, shown in the diagrams below. No additional classes were needed, as the remaining transport infrastructure is provided by the LiveKit SDK itself.



The first pair of types operates at the wire level. \texttt{TransportPacket} is a serializable envelope that wraps outgoing data before it is published to the LiveKit data channel. Its \texttt{Message} variant carries a small payload that fits within the 14\,KB limit in a single packet, while the \texttt{Chunk} variant carries a fragment of a larger message along with metadata --- a unique transfer identifier, the chunk index, and the total number of chunks --- so that the receiver can reassemble the original payload. \texttt{NetworkMessage} represents the application-level content inside each packet and has three variants: \texttt{Sync}, which carries raw Automerge synchronization bytes; \texttt{Chat}, which carries a plain-text chat string; and \texttt{Cursor}, which carries an $(x, y)$ coordinate pair for remote cursor rendering.

The second pair of types bridges the two application threads. \texttt{AppCommand} is sent from the UI thread to the background network thread through an MPSC channel. It can instruct the network thread to broadcast a \texttt{NetworkMessage} to all room participants (\texttt{Broadcast}), send a message to a specific subset of recipients (\texttt{Send}), or terminate the connection (\texttt{Disconnect}). Conversely, \texttt{AppMsg} is sent from the background thread back to the UI thread. It delivers incoming network messages (\texttt{NetworkMessage} variant with the sender identity attached), participant lifecycle events (\texttt{ParticipantConnected}, \texttt{ParticipantDisconnected}), and general log entries (\texttt{Log}). Together, these four types form a clean, minimal communication contract that decouples the UI rendering loop from all asynchronous network I/O.

\subsection{Cursor synchronization}

Remote cursor positions are synchronized separately from the CRDT document state. Unlike strokes, which are permanent drawing artifacts stored in the Automerge document and persist across sessions, cursor positions are ephemeral --- they represent a momentary state that is only meaningful while the user is actively interacting with the canvas. For this reason, cursors are not part of the CRDT data model and are instead transmitted as lightweight, fire-and-forget messages over the LiveKit data channel.

When the local user moves the mouse over the whiteboard canvas, the application checks whether at least 50 milliseconds have elapsed since the last cursor broadcast. This throttling mechanism prevents flooding the network with redundant updates during rapid mouse movements while still providing a smooth real-time experience for remote observers. If the throttle interval has passed, the current pointer coordinates are wrapped in a \texttt{NetworkMessage::Cursor\{x, y\}} and sent via \texttt{AppCommand::Broadcast} to all participants in the room.

On the receiving side, incoming cursor messages are stored in a \texttt{HashMap<String, Point>} keyed by the sender's identity. During each rendering frame, the UI iterates over this map and draws a colored circle at each remote user's last known position, accompanied by a label showing the user's identity. The color assigned to each remote cursor is deterministic --- it is derived by hashing the user's identity string --- so that the same user always appears with the same color across all peers' screens. When a participant disconnects, their entry is removed from the cursor map, and the indicator disappears from the canvas immediately.

\begin{figure}[H]
	\centering
	\includegraphics[width=0.99\textwidth]{./images/cursor-broadcasting.png}
	\caption{Broadcasting and rendering of remote cursor positions}
	\label{fig:cursor-broadcasting}
\end{figure}

This design keeps cursor synchronization simple and performant. Because cursor data is not persisted in the CRDT document, it adds no overhead to the Automerge sync protocol and does not increase the document size over time. The trade-off is that cursor positions are lost if a message is dropped, but given the high update frequency (up to 20 updates per second per user), any lost position is quickly replaced by a newer one, making occasional packet loss visually imperceptible.
